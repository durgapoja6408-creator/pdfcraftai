## Patterns established

1. **Idempotent Python wrap scripts** for mechanical refactors across many files.
   Examples: `wrap-set-error.py` (G1), `wrap-fetch-ai-retry.py` (M20),
   `wrap-uploaded-file-preview.py` (M18). Each script skips already-migrated files
   and warns on non-matching patterns, so they can be re-run safely.
2. **Static parse + assert** as a CI guard for cross-cutting invariants. Cheaper
   than runtime tests and catches the "developer forgot the new pattern" class
   of regression. See `scripts/test-*.mjs`.
3. **Hooks for one-shot mount effects** (`useHandoffConsumer`,
   `useFileUrlConsumer`, `useScrollErrorIntoView`) that consumers wire with one
   line. Each hook is single-purpose and composable.
4. **Module-level caches** (`lib/client/handoff.ts` registry, `useFirstPagePreview`
   LRU) backed by Maps with bounded size + LRU eviction. Survives Next.js
   client-side route changes; doesn't survive hard reload.

## Operational note — site 503 during high-velocity deploys

Around 17:30 UTC the site went 503 with the documented Hostinger LSAPI thread-cap
cascade pattern (see CLAUDE.md §5). ~25 commits in rapid succession triggered
auto-pull churn beyond what the cgroup thread budget could handle. SSH itself
became unreachable mid-handshake (the auth daemon was thread-starved too).

**Recovery procedure**:
1. Wait. The thread pool drains naturally — usually 5–15 minutes once new
   pushes stop landing.
2. If still 503 after 15 min: hPanel → Resource Usage → "Stop running process"
   manually. The app auto-restarts and the 503 clears.
3. **Do NOT** SSH-pkick repeatedly during a cascade — that adds load and
   prolongs the recovery. ONE pkick max per cycle (CLAUDE.md §5 wording).

The recent commits (post-`8453574`) are on `main` and will auto-pull once
the workers come back. No data risk; just availability.
